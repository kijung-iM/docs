## 形式

MXQLは、各行ごとに**コマンド**と**オペランド**で構成され、分かち書きで区分します。

```sql
<コマンド> <オペランド>
```

`コマンド`は、一つの単語からなる予約語です。`コマンド`は大文字で入力し、`オペランド`は小文字で入力します。コマンドごとに入力可能な`オペランド`の形式は決まっています。`オペランド`には4つのタイプの値が来ることがあります。

1.  オペランドがない場合

```sql
TAGLOAD
```

1.  文字列（数字または単語）

```sql
CATEGORY app_counter
```

1.  文字列配列

```sql
SELECT [ time, pcode ]
```

1.  JSON文字列型

```sql
FILTER { key : tx_count, value : 5}
```

```sql title='샘플 MXQL 퀴리'
CATEGORY app_counter
TAGLOAD
SELECT [ time, pcode ]
FILTER { key : tx_count, value : 5}
```

## 段階別構成

MXQLは、段階別構成を持っています。各段階別に使用できる**コマンド**の種類は決まっており、各段階の名前と特徴は次のとおりです。 

1.  **メトリクスの選択**：どのエージェントから収集したどのメトリクスを使用するかを選択してください。

2.  **メトリクス読み込み**：前の段階で設定した値を使用してメトリクスを読み込みます。ほとんどの場合は`TAGLOAD`を利用しますが、特殊な場合は`FLEXLOAD`を利用してください。`FLEXLOAD`を使用する必要がある場合は、[次の文書](mxql-loading#flexload-items)を参考にしてください。

3.  **メトリック加工**：前の段階で読み込んだメトリクスに対し、段階的に加工を行います。

```sql title='단계별 구성 예시'
# メトリクスの選択段階
CATEGORY app_counter -- カテゴリ選択

# メトリクスの読み込み段階
TAGLOAD -- データ1000件照会

# メトリクスの加工段階
SELECT [time, oid, active_tx_count, tx_count, tx_error] -- データ1000件のうち5フィールドのみ次の段階に伝達
FILTER {expr : "tx_count > 40"}        -- データ1000件のうち100件が通過
FILTER {expr : "active_tx_count > 10"} -- データ100件のうち10件が通過
FILTER {expr : "tx_error < 3"}        -- データ10件のうち3件が通過
```

以下は、メトリクス加工段階をすべて通過したメトリクスの例です。

![예시](https://img.whatap.io/media/images/mxql_metrics_calculate_example.png)

## コメント

"#"または"--"で始まる文章は無視されます。

```sql title='주석 예시'
# データ照会設定
CATEGORY app_counter

# データ照会
TAGLOAD

# データ加工
SELECT [ time, pcode ]
FILTER { key : tx_count, value : 5}
```

## MetricValue(複合値) {#metric-value}

Metric Value(複合値)は、メトリクス加工段階でよく使われる演算に対応するMXQLの資料構造です。メトリクス加工段階の[GROUP](mxql-calculate#group)、[UPDATE](mxql-calculate#update)コマンドは、メトリクスがMetricValue形式で保存されている場合に限って使用できます。

たとえば、次のようなデータがあるとします。 

| time                | tx_count |
| ------------------- | -------- |
| 2021/06/24 13:40:00 | 1        |
| 2021/06/24 13:40:10 | 2        |
| 2021/06/24 13:40:20 | 3        |
| 2021/06/24 13:40:30 | 4        |
| 2021/06/24 13:40:40 | ５        |
| 2021/06/24 13:40:50 | 6        |

上のデータを30秒間隔で[GROUP](mxql-calculate#group)のmergeオプションを進めると、次のような形にデータを変形することができます。

| time                                      | tx_count                 |
| ----------------------------------------- | ------------------------ |
| 2021/06/24 13:40:00 ~ 2021/06/24 13:40:20 | 1、2、3に対するMetric Value    |
| 2021/06/24 13:40:30 ~ 2021/06/24 13:40:50 | 4、5、6に対するMetric Value    |

データをMetricValueに変換すると、計6つのオプションが使用できます。

| オプション | 機能                                                      |
| ----- | ------------------------------------------------------- |
| sum   | MetricValueに含まれる値を追加します。                                |
| min   | MetricValue値に含まれる最小値を検索します。                             |
| max   | MetricValue値に含まれる最大値を検索します。                             |
| last  | MetricValue値に含まれる、最後に追加された値を戻します。                       |
| avg   | MetricValue値に含まれる値の平均を求めます。                             |
| cnt   | MetricValue値に保存されている値の数を検索します。                          |

MetricValueオプションは、[UPDATE](mxql-calculate#update) コマンドを通じて利用できます。

```sql title='UPDATE'
CATEGORY app_counter
TAGLOAD
SELECT [ time, okindName, okind, apdex_satisfied, apdex_tolerated, apdex_total]
-- GROUPコマンドのmergeオプションを通じてMetric Valueに変換するfieldを設定
GROUP { timeunit:5000, pk:okind, last:okindName, merge:[apdex_satisfied, apdex_tolerated, apdex_total] }
-- UPDATEコマンドを通じてsumオプションを適用
UPDATE { key:[apdex_satisfied, apdex_tolerated, apdex_total], value:sum }
```

## MetricValueタイプのデータ利用方法

-   `GROUP`コマンドのmergeオプションにフィールドを設定します。

    ```sql
    CATEGORY app_counter
    TAGLOAD
    SELECT [ time, okindName, okind, apdex_satisfied, apdex_tolerated, apdex_total]
    -- GROUPコマンドのmergeオプションを通じてMetric Valueに変換するfieldを設定
    GROUP { timeunit:5000, pk:okind, last:okindName, merge:[apdex_satisfied, apdex_tolerated, apdex_total] }
    -- UPDATEコマンドを通じてsumオプションを適用
    UPDATE { key:[apdex_satisfied, apdex_tolerated, apdex_total], value:sum }
    ```

-   収集サーバーにデータを保存するとき、すべてのフィールドにはMetricValue形式で保存されたカテゴリがあります。**_サイトマップ_** > **_分析_** > **_メトリクス照会_** > **_カテゴリ_**オプションには、**_デフォルト_**、**_5분_**、**_1時間_**単位で選択できるカテゴリを確認できます。ここで、**5分**または**1時間**を選択できるカテゴリは、MetricValue形式で保存されたカテゴリです。

    ![MetricValue形式として設定できるカテゴリ一覧](https://img.whatap.io/media/images/mxql-3.png)

    **_5分_**または**_1時間_**を選択できるカテゴリの名前に`{m5}`または`{h1}`を組み合わせると`GROUP`コマンドのmergeオプションを適用せず、すぐに`UPDATE`コマンドの`sum`オプションを適用できます。 

    ```sql title='모든 필드가 MetricValue타입으로 저장된 app_counter{m5} 카테고리를 사용'
    CATEGORY app_counter{m5}
    TAGLOAD
    SELECT [time, pname, host_ip, pid, httpc_count]
    -- GROUPコマンドを適用しなくても、すでにデータがMetricValue型なのでUPDATEコマンドを適用できます。
    UPDATE { key : httpc_count, value : avg }
    ```

## MetricValue型の基本演算`avg`

MetricValue型フィールドの基本出力形態は`avg`です。つまりMetricValue型のフィールドは、別のオプションを設定しないと`avg`を適用します。次の2つのクエリは同じ結果を出します。

```sql title='avg를 지정하지 않은 경우'
CATEGORY app_counter
TAGLOAD
SELECT [time, pcode,pname, tps]
GROUP  {timeunit:5000, pk:pcode, last: pname, merge:tps}
```

```sql title='avg를 지정한 경우'
CATEGORY app_counter
TAGLOAD
SELECT [time, pcode,pname, tps]
GROUP  {timeunit:5000, pk:pcode, last: pname, merge:tps}
UPDATE {key:tps, value:avg}
```

## 事前定義MXQLクエリ文{#custom-mxql}

MXQLクエリを直接作成するのではなく、事前に定義されたMXQLクエリファイルのパスを設定してMXQLを実行することができます。例えば、'エージェント別アクティブTX件数'、'\<区間別>件数'、'直近15秒'を求めるMXQLクエリは次のとおりです。

```sql title='에이전트별 액티브TX 건수, \<구간별> 건수, 최근 15초'
HEADER {  act0$:I, act3$:I, act8$:I, act$:I}
TIME-RANGE {duration:15s, etime:$etime}
OIDSET {oid:$oid, okind:$okind, onode:$onode}
CATEGORY app_counter

TAGLOAD {backward:true}

SELECT [oid, oname, active_tx_0, active_tx_3, active_tx_8, active_tx_count, pcode]
FIRST-ONLY {key:oid}
RENAME {src:active_tx_0,  dst:act0}
RENAME {src:active_tx_3,  dst:act3}
RENAME {src:active_tx_8,  dst:act8}
RENAME {src:active_tx_count,  dst:act}
CREATE {key:_id_, from:oid}
CREATE {key:_name_, from:oname}

INJECT default
```

もし上記のクエリが収集サーバーに登録されている場合は、次のように入力するだけで同じデータを照会することができます。設定したパスに保存したサーバーのファイル内容を読み込んで呼び出す方式です。

```txt title='사전 정의한 파일의 경로를 입력합니다.''
/app/act_tx/act_tx_oid
```

[次の文書](#custom-query-saas)で使用可能なクエリを確認できます。

## 参考資料

### バインド変数(パラメータ){#bind-parameter}

MXQLではバインド変数を使用できます。バインド変数は`$`で始まります。また、valueに該当する部分のみを使用することもできます。 

```sql
SKIP $skip_value
```

```sql
SKIP [$skip_value]
```

```sql
SKIP {value:$skip_value}
```

```sql title='불가능한 경우 : key로 바인드 변수를 전달할 수 없습니다.'
SKIP {$option:10}
```

クエリでバインド変数を使用した場合は、MXQLを実行する際に入力する値を伝える必要があります。

![バインド変数設定の例](https://img.whatap.io/media/images/mxql-1.png)

:::note
バインド変数の名前は、大文字と小文字のアルファベットのみ可能です。数字および特殊文字は、バインド変数の名前に含めることはできません。 
:::

### データの読み込み方式

MXQLで照会できるデータは、[メトリクス](../metrics/common-metrics-intro)の形式によって大きく2つに分けることができます。

-   メトリクスのデータがタグとフィールドに分けて保存されたデータ([TAGLOAD](mxql-loading#tagload)を使って読み込みできるデータ)
-   メトリクスにすべてのデータがフィールドに保存されているデータ([FLEXLOAD](mxql-loading#flexload)を使って読み込みできるデータ)

ほとんどのカテゴリは`TAGLOAD`を使います。[次の文書](mxql-loading#flexload-items)に含まれるカテゴリのデータを使用する場合に限って`FLEXLOAD`を使います。

## SaaSサービスで提供する事前定義されたMXQLクエリの一覧{#custom-query-saas}

MXQLクエリを直接作成せずにパスに設定できる機能の主な目的は、複雑なクエリを簡単に呼び出すためではなく、管理者が直接自分の意図通りクエリを作成し、これを使用することです。したがって、すでにYardにどのようなクエリが含まれているかを確認して使用するよりは、直接クエリを登録する方向で活用する必要があります。`JOIN`コマンドを使用する場合は、MXQLクエリを使用する中でも特別な場合であるため、管理者が直接クエリを登録し、そのファイルのパスを使用することになっています。

:::note
_yard.conf_ファイルの`mxql_root`に設定したパスに登録したいクエリをファイルとして保存できます。(default ./mxql)
:::

### エージェント別アクティブTX件数、件数、直近15秒

-   パス：_/app/act_tx/act_tx_oid_
-   クエリ： 

    ```sql
    HEADER {  act0$:I, act3$:I, act8$:I, act$:I}

    TIME-RANGE {duration:15s, etime:$etime}

    OIDSET {oid:$oid, okind:$okind, onode:$onode}

    CATEGORY app_counter

    TAGLOAD {backward:true}

    SELECT [oid, oname, active_tx_0, active_tx_3, active_tx_8, active_tx_count]
    FIRST-ONLY {key:oid}
    RENAME {src:active_tx_0,  dst:act0}
    RENAME {src:active_tx_3,  dst:act3}
    RENAME {src:active_tx_8,  dst:act8}
    RENAME {src:active_tx_count,  dst:act}

    CREATE {key:_id_, from:oid}
    CREATE {key:_name_, from:oname}
    ```

### エージェント別詳細情報 & エージェント別アクティブTX件数、件数、直近15秒

-   パス：_/app/act_tx/agent_with_tx_
-   クエリ： 

    ```sql
    CATEGORY agent_list

    OIDSET {oid:$oid, okind:$okind, onode:$onode}

    FLEXLOAD

    JOIN {query:'/app/act_tx/act_tx_oid', pk:oid, field:[act0,act3,act8, act] }

    UPDATE {key:act0, notnull:0}
    UPDATE {key:act3, notnull:0}
    UPDATE {key:act8, notnull:0}
    UPDATE {key:act, notnull:0}

    RENAME {src:[act0, act3, act8, act],  dst:[normal, slow, verySlow, total]}

    INJECT default
    ```
